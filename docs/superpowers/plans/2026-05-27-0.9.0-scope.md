### 4. Per-site lighthouse URL via `package.json#reddoor.lighthouseUrl` _(NEW — discovered 2026-05-27 smoke-testing CalTex)_

**Problem:** [src/configs/lighthouse.ts:4](../../../src/configs/lighthouse.ts#L4) and [src/configs/playwright-a11y.ts](../../../src/configs/playwright-a11y.ts) both hardcode `http://localhost:5173/dev/a11y-fixtures`. That route exists on the original Reddoor sites (hand-crafted at the start of the fleet) but no recipe in this repo creates it — so any newly-onboarded site (e.g. CalTex) fails the lighthouse audit with "no manifest written (exit 1)". The audit framework itself works; the URL just resolves to a 404.

**Surface:**

```jsonc
// caltex-landing/package.json
{
  "name": "caltex-landing",
  "reddoor": {
    "lighthouseUrl": "http://localhost:5173/",
  },
}
```

**Resolution:**

1. `src/audits/lighthouse.ts` reads `package.json#reddoor.lighthouseUrl` from `site.path` before writing the lhci config.
2. Falls back to the current `/dev/a11y-fixtures` URL if the field is missing — keeps every existing Reddoor site working without edits.
3. Same field applies to `src/configs/playwright-a11y.ts` for consistency.
4. Document in README under the onboarding section: "new sites without `/dev/a11y-fixtures` should set `package.json#reddoor.lighthouseUrl` to the page lhci should audit (homepage is fine)."

**Test plan:**

- Unit: lighthouse audit reads the override when the field is present, falls back when absent.
- Unit: malformed JSON / missing package.json doesn't crash — falls back to default.
- Regression: existing fixture sites (`pristine-starter`, `drifted-configs`) keep working unchanged.

**Why not Airtable:** Adding a Lighthouse URL column to the Websites table would couple `audit` to network on every run — currently audits can run fully offline against a local checkout. Per-site `package.json` keeps that boundary.

**Why not `.reddoor.json`:** Introduces a new file in every onboard for a single field. If 0.9.0+ grows the per-site config surface significantly, splitting out into `.reddoor.json` (or `reddoor` key in `package.json` continuing to grow) is a future migration.

**Dependency on item 3 (`init`):** If `init` ends up generating an a11y-fixtures route on every new site, item 4 becomes "nice to have" rather than "required to onboard." Either order works.
