## Findings — HIGH

**HIGH-1 — `verifyTurnstile` collapses "expired/duplicate token" into "fail", so slow or double-clicking *human* submitters get auto-filed as spam. Now live on 6 sites.** [turnstile.ts:52](../../src/forms/turnstile.ts#L52) reads only `obj.success` and returns `"fail"` for every `success:false` — but Cloudflare returns `success:false, error-codes:["timeout-or-duplicate"]` for a token that **expired** (300s TTL — a user filling a long form) or was **re-submitted** (double-click, back-then-resubmit). Downstream: [spam-classifier.ts:116-117](../../src/forms/spam-classifier.ts#L116) adds `+70` for a `"fail"`, [spam-classifier.ts:121-123](../../src/forms/spam-classifier.ts#L121) adds `+30` per URL, `SPAM_THRESHOLD = 100`, and [ingest.ts:109](../../src/forms/ingest.ts#L109) buckets `>= 100` as `spam_auto`. So **fail (+70) + one pasted URL (+30) = exactly 100 → `spam_auto`** — and [spam-classifier.test.ts:197-205](../../tests/forms/spam-classifier.test.ts#L197) asserts that 100 boundary *as intended behavior*. The lead isn't hard-lost (it lands in the `spam_auto` review queue) but the operator notification is suppressed and it's mis-classified. **Amplifier:** on a `requireTurnstile` site, [ingest.ts:110](../../src/forms/ingest.ts#L110) forces `spam_auto` on *any* `"fail"` regardless of content — so once you check that box (your own next-step), every expired/duplicate legit submit is silently spam-bucketed. **Fix:** parse `error-codes` in `verifyTurnstile`; map operational/benign codes (`timeout-or-duplicate`, `internal-error`) → `"unverifiable"` (fail-open), reserve `"fail"` for `invalid-input-response`. Secondary: reconsider the `+70` weight — at 70% of threshold, one benign co-signal (a URL) tips it over. _(Confidence: high — every hop traced and confirmed against code + the asserting test.)_

**HIGH-2 — No test pins the highest-blast-radius fail-open invariant: `requireTurnstile` + `unverifiable` must stay `new`.** The code is correct today — [ingest.ts:110](../../src/forms/ingest.ts#L110) gates escalation on `turnstile === "fail"`, so a **Cloudflare siteverify outage** (or a JS-off user with no token) yields `"unverifiable"` → stays `new` and leads keep flowing on gated sites. But the suite tests only `requireTurnstile + "fail" → spam_auto` ([ingest.test.ts:252](../../tests/forms/ingest.test.ts#L252)), never `requireTurnstile + "unverifiable" → new`. A plausible future "tighten it" refactor (`=== "fail"` → `!== "pass"`) would silently route **every lead on every gated site into `spam_auto` during a Cloudflare outage**, with green tests. Given HIGH-1's fix touches this exact area, add the guardrail test in the same PR: `requireTurnstile:true` + `turnstile:"unverifiable"` + clean verdict → `createSubmission` called with `status:"new"` + `notify` invoked. _(Confidence: high — invariant location confirmed; test-gap confirmed by reading the describe block.)_
