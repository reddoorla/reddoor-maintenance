## Top of stack (do these first)

So a night where 9 sites flake and 1 writes is a green checkmark over a 90%-stale dashboard. This isn't hypothetical: **erp-industrials already fails every fleet run** (HIGH-D below), so the nightly job is _already_ silently partial. Fix: key the job on the **absence** of the `⚠ … not written` line (the data is already in `FleetWriteResult.failed`), or assert a count threshold against expected fleet size. **While you're in the file, add `timeout-minutes: 30`** — there's none today (HIGH-A note 2). This is the highest-leverage item and it's the thing you flagged.
2. **Add a `pretest` build guard (~20 min, [package.json](../../package.json)).** `pretest: NONE` confirmed; `test` is bare `vitest run`, and the CLI tests `execFileSync` the compiled `dist/cli/bin.js`. So `pnpm test` is only correct if `dist/` happens to be fresh. **This exact gap, flagged as M6 in the 2026-06-05 brief and never fixed, bit us today** — the M2.2a final review caught a guard-test that passed locally against a stale `dist` and would have failed CI. Add a `pretest` hook (`pnpm build`) or a vitest `globalSetup` that builds once. This one has now cost real time twice; stop it.
3. **Make `cloneIfNeeded` derive a clone URL from `gitRepo` (~30 min, [src/cli/fleet/clone-if-needed.ts:65](../../src/cli/fleet/clone-if-needed.ts#L65)).** As of today's M2.2a, the Airtable inventory sets `deployedUrl` + `gitRepo` and **deliberately no longer sets `repoUrl`**. Lighthouse-only fleet runs are fine (they use `deployedUrl`). Fix: `const repoUrl = site.repoUrl ?? (site.gitRepo ? \`https://github.com/${site.gitRepo}.git\` : undefined)`, validate `gitRepo`matches`^[\w.-]+/[\w.-]+$`before interpolating, and run it through the existing`assertSafeRepoUrl`. (I flagged this as a carry-forward at merge; two independent reviewers tonight confirmed it HIGH and broader than I'd framed.)

## Findings — HIGH

**HIGH-A — Nightly workflow goes green on a partial outage; no job timeout.** [.github/workflows/fleet-lighthouse.yml:57](../../.github/workflows/fleet-lighthouse.yml#L57). Covered in Top-of-stack #1. **HIGH-B — No `pretest` build guard → `pnpm test` trusts a possibly-stale `dist`; it bit today.** [package.json](../../package.json) (`pretest: NONE`). Covered in Top-of-stack #2. This was **M6 in [MORNING_REPORT_2026-06-05.md](MORNING_REPORT_2026-06-05.md):71** — "CLI tests exec the built dist with no pretest build guard → spurious failures." It was never acted on, and today the M2.2a per-task reviews all passed a `--write-airtable`+`--fleet` guard test against a stale `dist` that encoded the _old_ behavior; only the final holistic review (which rebuilt `dist`) caught it. A recurring, predicted, now-twice-costly trap.

**HIGH-C — `cloneIfNeeded` can't clone Airtable fleet sites (reads `repoUrl`, inventory now sets only `gitRepo`).** [src/cli/fleet/clone-if-needed.ts:65](../../src/cli/fleet/clone-if-needed.ts#L65) + [src/inventory/airtable.ts:42](../../src/inventory/airtable.ts#L42). Also mirror the fix into the JSON inventory provider ([src/inventory/json.ts:24](../../src/inventory/json.ts#L24)), which likewise drops `gitRepo`/`deployedUrl` (MEDIUM-G).

**HIGH-D — `spawn` timeout kills only the direct child, orphaning Chrome/vite — recreating the zombies the audits fight.** [src/audits/util/spawn.ts:24](../../src/audits/util/spawn.ts#L24),[38](../../src/audits/util/spawn.ts#L38). The child is spawned **without `detached: true`**, so `child.kill("SIGTERM")` signals only the `npx`/`pnpm` wrapper — not the grandchildren (vite via `npm run vite:dev`, Chromium under lhci/playwright). On the 5-min timeout, `npx` dies but **vite keeps squatting its port and Chrome keeps running**. The irony: `findFreePort` + `--strictPort` were added specifically to dodge "zombie vite on 5173," and the timeout path is exactly what _creates_ those zombies — in a repo whose own self-updating CI checks for a clean tree. Fix: `{ detached: true }` + `process.kill(-child.pid, "SIGTERM")` on timeout, with a `SIGKILL` escalation after a grace delay; `clearTimeout` on `close`/`error`. _(Two reviewers independently; the workflow review predicted it, the audit-internals review confirmed it in code.)_

## Findings — MEDIUM

**MEDIUM-A — `deps` audit measures the declared `package.json` range, not the installed version — the drift number tracks the wrong thing.** [src/audits/deps.ts:21-71](../../src/audits/deps.ts#L21). The audit never reads the lockfile, never runs `pnpm outdated`, never inspects `node_modules`; it `stripCaret`s the authored range and compares to the baseline range. So a site with `"svelte": "^5.0.0"` but `5.55.10` actually installed reports `drift: "minor"` against baseline `^5.55.10` even though the install matches. This is the "Deps Drifted" number the operator sees on the dashboard, so it drives wrong decisions. **Decide intent:** if "declared floor" is the goal, fix the summary wording ("declared range"); if real drift, read the lockfile or `pnpm outdated --format json`.

**MEDIUM-B — Dashboard client-token model is still load-bearing AND inconsistent, contradicting the decided retirement.** [netlify/functions/site-dashboard.mts](../../netlify/functions/site-dashboard.mts) + [src/dashboard/auth.ts](../../src/dashboard/auth.ts) + [src/dashboard/fleet-render.ts:63](../../src/dashboard/fleet-render.ts#L63). The roadmap ([fleet-scale-roadmap.md:160](../superpowers/specs/2026-06-02-fleet-scale-roadmap.md)) says retire the token model entirely (one operator password). Reality: `/s/:slug` is gated **solely by `verifyDashboardToken`** (no password), `/` is gated **solely by the password** (no token) — two inconsistent auth models live at once — and the password-gated fleet homepage **embeds every site's secret token** in the rendered `?t=` links. Not externally exploitable (you need the password to see the homepage), but it's an architectural exposure + the token-disclosure surface. **Decide:** execute the retirement (drop `verifyDashboardToken`, gate `/s/:slug` with basic-auth, make `dashboardToken` a plain visibility flag), or update the roadmap to say "deferred." Code and spec currently disagree.

**MEDIUM-C — `sync-configs` will strip security headers from a live site's `netlify.toml`.** [src/recipes/sync-configs/templates.ts:125](../../src/recipes/sync-configs/templates.ts#L125). The canonical `netlify` template ships only `command`/`publish`/`functions`/`NODE_VERSION` — **none of the starter's real security headers** (`Strict-Transport-Security`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`, COOP, immutable cache-control). Because `netlify` is in the default sync set and uses **exact-match overwrite** with no compliance carve-out (unlike `svelte`, which got one), a plain `sync-configs` run **deletes those headers** and downgrades `pnpm run build`→`pnpm build`. Same class as the svelte-clobber bug that carve-out was added to fix. **Fix:** make `netlify` opt-in (`--only netlify`), or add a "leave alone if it already has `[[headers]]`" guard, or bring the template up to the starter. **Check live fleet `netlify.toml` files for already-flattened headers from a prior sync.**

**MEDIUM-D — a11y audit's temp dirs live inside the git checkout and orphan on a hard crash (recurred from 06-05 M3).** [src/audits/a11y.ts:178](../../src/audits/a11y.ts#L178),[186](../../src/audits/a11y.ts#L186). `mkdtemp` writes `.reddoor-a11y-spec-*/` and `.reddoor-a11y/` **inside `site.path`** (required for module resolution). They're cleaned on the normal/ENOENT paths but not if the parent process is timeout-killed mid-run, leaving untracked files in a client repo whose self-updating CI checks for a clean tree. **This was M3 in the 2026-06-05 brief** ("orphaned `.reddoor-a11y-spec-*` showed up uncommitted") — still open. Fix: wrap creation→cleanup in `try/finally`, and add `.reddoor-a11y-spec-*/` to the root `.gitignore` **and** `CANONICAL_GITIGNORE_ENTRIES` so the fleet inherits it.

**MEDIUM-E — Airtable write is all-or-nothing on Lighthouse: a Lighthouse miss discards good a11y/deps/security data for that site.** [src/audits/write-audits-to-airtable.ts:46-53](../../src/audits/write-audits-to-airtable.ts#L46). `hasRealScores(lhResult)` false → throws `exitCode 1` → the a11y/deps/security writes at L66-85 never run. So a Lighthouse Chrome-timeout (exactly erp's nightly fate) throws away three other valid audits' results for that site. Given Lighthouse is the most timeout-prone audit, this coupling is worth reconsidering: write the non-Lighthouse counts before the Lighthouse gate, or downgrade a Lighthouse miss to a per-audit skip rather than a whole-row abort.

**MEDIUM-F — erp-industrials' reproducible flake is most likely the 3-min deployed-Lighthouse timeout, not concurrency.** [src/audits/lighthouse.ts:240](../../src/audits/lighthouse.ts#L240). `deployedLighthouse` runs `numberOfRuns: 3` desktop passes under one `3 * 60_000` wall-clock — the checkout path uses `5 * 60_000`. Three serial cold runs of a slow site (lhci's own `maxWaitForLoad` ~45-60s each) can plausibly exceed 180s → SIGTERM → no `lhr-*.json` → no scores → filed under `failed`. That matches "no scores in fleet runs but P75 in an isolated, warm-cache, full-budget spot-check." **To confirm:** re-run erp alone with `--concurrency 1` and a 6-min timeout and watch whether it crosses 180s. **Fix:** bump `deployedLighthouse` timeout to `5 * 60_000` (match checkout) and/or set explicit `settings.maxWaitForLoad`; and surface `raw.stderr` into the workflow log so the next unattended miss is diagnosable without a re-run.

**MEDIUM-G — JSON inventory provider drops `gitRepo`/`deployedUrl`.** [src/inventory/json.ts:24](../../src/inventory/json.ts#L24). `validate()` copies `path`/`name`/`repoUrl`/`meta` only — so the `--fleet sites.json` fallback can't carry a `gitRepo`-only or `deployedUrl` entry. Mirror HIGH-C's fix here once it lands. Low blast radius today (Airtable is the live source) but it's latent drift between the two providers.

**MEDIUM-H — `getWebsiteBySlug` does a full Websites-table scan per request.** [src/reports/airtable/websites.ts:133](../../src/reports/airtable/websites.ts#L133). Every `/s/:slug` hit pages the entire table then `.find()`s in memory; same for the fleet homepage. Fine at ~12 rows, but it's an unbounded fetch with no cache — a rate-limit/latency risk if `/s/:slug` ever sees real traffic. Fix: `filterByFormula` for the single row, or a short-TTL cache.

**MEDIUM-I — No alerting on a red nightly run.** Beyond GitHub's best-effort email to the workflow's last-pusher (per-account, easily muted), there's no Slack/webhook/issue-on-failure. Combined with HIGH-A, the practical state is: a _partial_ outage is silently green, and a _total_ outage pings one person if their notifications are on. Add an `if: failure()` step that opens/updates a tracking issue. (Pairs with fixing HIGH-A.)
